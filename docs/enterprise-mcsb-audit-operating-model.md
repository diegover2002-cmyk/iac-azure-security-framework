# Enterprise MCSB Audit Operating Model

> Purpose: define how the organization audits, evidences, and future-automates MCSB-aligned controls for Azure services, cross-cutting domains, and CI/CD security gates.
> Audit basis: repository control matrix, normalized catalog, cross-cutting domain guidance, official Checkov documentation, and current Checkov rule inventory.
> Date: 2026-03-23

## Executive Summary

This operating model covers **232** documented controls across **31** deployable Azure service families and **4** cross-cutting domains.

### Current State by Definition Status

| Status | Controls | Meaning |
|---|---:|---|
| Implemented | 158 | Control is documented in a Golden Reference or Strong Baseline service, or in established cross-cutting guidance. |
| Partial | 74 | Control exists but needs deeper service detail, stronger evidence modeling, or better automation coverage. |
| Pending | 0 | Control family or domain is identified but still lacks row-level definition in the repository. |

### Scope Separation

- Deployable Azure services are audited from `controls/MCSB-control-matrix.md` and their service-specific `controls.md` files.
- Cross-cutting domains are audited as organizational controls, not as single Terraform resources.
- Checkov is referenced where it helps automate review, but it does not define the organization's control intent.
- `Endpoint Security` and `AI Security` are recognized cross-cutting domains, but they are still pending row-level control definitions in this repository.

## Audit Decision Model

| Decision Area | Organizational Rule |
|---|---|
| Source of control intent | This repository's matrix and service/domain baselines |
| Source of Checkov tool behavior | Official Checkov documentation at `checkov.io` |
| Source of rule existence and current IDs | `bridgecrewio/checkov` repository or pinned internal mirror |
| Source of enterprise audit outcome | This operating model, using repository intent plus validated tool and platform evidence |
| Blocking policy | `Must` controls with direct or approved custom automation should block CI/CD |
| Manual review policy | `Must` controls without safe automation still require explicit reviewer sign-off |
| Evidence policy | Every control must define the minimum evidence needed to record pass or fail |

## Audit Method Categories

| Category | Meaning | Typical Evidence |
|---|---|---|
| Direct | Control can be evaluated in CI/CD using Checkov or another deterministic scanner against repository content. | Scan output, SARIF, policy reports, Terraform plan review. |
| Partial | Control has some automatable elements, but still needs Azure configuration review, logs, or runtime evidence. | Terraform plus diagnostics, policy assignment state, role assignments, exported settings. |
| Manual | Control is primarily operational or inherited and requires governance evidence or platform review. | Audit logs, review records, platform screenshots, exported configuration, attestation. |

## Gap Summary

| Gap Type | Current Observation | Recommended Action |
|---|---|---|
| Pending cross-cutting domains | `Endpoint Security` and `AI Security` are recognized but still lack row-level control catalogs. | Create dedicated `controls/<domain>/controls.md` guidance or an equivalent domain catalog under `docs/`. |
| Catalog coverage gap | The normalized service control catalog still covers fewer rows than the full deployable matrix. | Expand `controls/MCSB-service-control-catalog.md` in maturity order, starting with strong-baseline services. |
| Traceability inconsistency gap | Some repository Checkov references and service-level mappings still need normalization across matrix, catalog, and per-service baselines. | Continue the Checkov normalization pass and treat this operating model as the enterprise audit layer that records the approved audit method even when tool mappings are still being reconciled. |
| Automation coverage gap | Many `Must` controls remain `Partial` because they require logs, diagnostics, RBAC, or policy-state evidence. | Define a future security-check engine that combines Terraform parsing, Checkov, and Azure configuration evidence. |
| Custom policy gap | Some controls are deterministic in Terraform but have no clean Checkov mapping today. | Track them as custom policy-as-code candidates in the future blocking check roadmap. |
| Operational evidence gap | Inherited or runtime controls such as platform-managed encryption and review processes need standard evidence templates. | Standardize pass/fail evidence records for manual and semi-automated controls. |

## Recommended Roadmap

1. Normalize the MCSB service control catalog so every Strong Baseline service has service-to-control rows suitable for export and CI/CD design.
2. Add row-level control definitions for `Endpoint Security` and `AI Security` as cross-cutting domain guidance.
3. Mark `Must` controls as `Direct`, `Partial`, or `Manual` in a future machine-readable export to support the blocking security check design.
4. Prioritize custom policy development for deterministic Terraform controls that lack a stable Checkov mapping.
5. Define a standard evidence contract for manual and semi-automated controls so the future validation tool can report `pass`, `fail`, and `needs evidence` consistently.

## Notes for Future Security Check Design

- Use this document as the enterprise operating model for how controls are audited, not as a replacement for service-specific guidance.
- Treat Checkov references as optional accelerators for automation, not as the sole definition of compliance.
- Where repository mappings are being normalized, prefer the audit method and evidence model in this document over a stale rule ID copied from an older baseline.
- For local or remote repository scanning, the future tool should combine repository parsing, Checkov output, and a policy layer for custom controls and manual evidence states.
